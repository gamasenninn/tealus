## 第5部: TTS 読み上げ機能の技術的深度

### パイプライン

```
ユーザーがメッセージ送信
  ↓ webhook
agent-server が回答生成
  ↓ botApi.pushMessage()
テキストメッセージ送信（即座にチャットに表示）
  ↓ fire-and-forget
Aivis Cloud API で音声合成（0.3-0.5秒）
  ↓ WAV
ffmpeg で Opus RTP に変換
  ↓ PlainTransport
mediasoup Router
  ↓ WebRTC
ブラウザのトランシーバーで受信・再生
```

### 技術的知見

1. **PlainTransport + comedia** — WebRTC スタックなしで mediasoup に音声を注入
2. **Consumer セットアップ待ち** — Producer 作成後 2 秒待ちが必要（頭切れ防止）
3. **ffmpeg adelay + apad** — 先頭/末尾の無音追加で音声品質改善
4. **ffmpeg 終了後の 1.5 秒待ち** — 最後の RTP パケットの処理待ち
5. **React useEffect + setTimeout** — 依存配列に毎レンダー変わるオブジェクトを入れるとタイマーが壊れる。ref パターンで解決
6. **express.json() と proxy の順序** — body parser が proxy より先だと PUT body が消費される
7. **fire-and-forget パターン** — TTS エラーがメッセージ送信に影響しない設計
8. **キュー管理** — 同時読み上げ防止の FIFO キュー

### 設計哲学: 声はエージェントの人格の一部

- 声の設定は room_settings.json — プロンプトと同じレイヤ
- ルームごとに異なる声 = ルームごとに異なるエージェントの人格
- ダッシュボードで設定 = エージェントの性格定義の一環
- 将来: 語尾「だじょ」のプロンプト + ひょうきんな声 → 性格と声の一体化

## 第6部: 発見された設計課題と次のフロンティア

### 設計課題: ダッシュボードの権限モデル（#156）

TTS 機能の実装過程で、ダッシュボードの根本的な設計課題が浮上：

1. **admin 限定** — 一般ユーザーがエージェント設定を変更できない
2. **DM vs グループ** — DM ではユーザーが自由に設定すべき、グループはガバナンスが必要
3. **ルーム一覧の視点** — 誰がログインしているかでルーム一覧が変わる

**解決方針（#156 で実装予定）:**
- チャット画面にエージェント設定パネルを追加
- DM: ユーザーが自由に設定可能
- グループ: ルーム作成者 or admin のみ
- ダッシュボード: admin のシステム管理 + オーバーライド

### 総評

**Tealus の AI は声を得た。**

前回のレポートで「AI と人間が対等に議論し、意思決定している」と書いた。今回はそれを超えた。

AI がテキストで応答し、同時にトランシーバーから声で語りかける。ユーザーは設定を ON にするだけで、トランシーバーは自動接続され、読み上げ後に自動切断される。ルームごとに異なる声を持ち、声はエージェントの性格の一部として管理される。

- PlainTransport で mediasoup に外部から音声を注入する技術基盤を確立

- fire-and-forget パターンにより、メッセージ送信のパフォーマンスに影響なし

**設計的には:**
- 「声はエージェントの人格の一部」という思想が確立
- room_settings.json にプロンプトと声を統合 → エージェントの性格定義が一箇所に
