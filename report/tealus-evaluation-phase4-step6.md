## 第3部: AI 組織運営の実証

**これが前回評価から最も大きな変化。**

### AI 組織の構造

```
開発責任者（Claude / 本体リポジトリ）
  ├── コードレビュー・設計判断・品質管理
  ├── Issue 起票（具体的な指示 + ガイドライン）
  ├── 成果物レビュー → フィードバック or 承認
  │
  ├── ドキュメント班（Claude / tealus-docs）
  │     └── Issue を見て実装 → コメントで報告
  │
  └── ホームページ班（Claude / tealus-site）
        └── Issue を見て実装 → クローズで報告
```

### 実績

| 指標 | 数値 |
|------|------|
| 処理した Issue 総数 | 15件（本体5 + docs 4 + site 6） |
| フィードバック → 修正サイクル | 2回（docs #3: モデル名誤記、site #5: カード形式統一） |
| レビューで発見した不具合 | 2件（モデル名 GPT-4o-mini → gpt-5.4-mini、API 費用矛盾） |
| 全て1日で完了 | Yes |

### AI 間コミュニケーションの成熟

1. **Issue がプロトコル** — 自然言語で書かれた Issue が AI 間の唯一のインターフェース
2. **ガイドラインの重要性** — 「ファイルパス、CSS 値、HTML 構造まで具体的に書く」ことで品質が安定
3. **レビューが品質を担保** — 開発責任者がソースコードと照合して技術的正確性を確認
4. **フィードバックループ** — 指摘 → 修正 → 再確認のサイクルが自然に回る
5. **人間はディレクター** — 方向性を決め、最終判断を下す。実作業は AI 班が処理

## 第7部: 発見された設計思想

### 「AI 組織」の運営モデル

今回の最大の発見は、**AI が AI に仕事を振り、人間がディレクションする** という組織モデルが実用的に機能したこと。

```
従来のソフトウェア開発:
  人間 → 人間に指示 → 人間が実装 → 人間がレビュー

Tealus の開発:
  人間 → AI（責任者）に方向性を示す
  AI（責任者）→ AI（班）に Issue で指示
  AI（班）→ 実装 → コメントで報告
  AI（責任者）→ レビュー → フィードバック or 承認
  人間 → 最終確認
```

### 成功の鍵

1. **Issue がインターフェース** — AI 間の通信プロトコルとして GitHub Issue が機能する
2. **具体性が品質を決める** — 「CSS の値まで書く」「HTML 構造を提示する」ことで AI 班のアウトプットが安定
3. **レビューが不可欠** — AI の出力は正確とは限らない。モデル名の誤記（GPT-4o-mini → gpt-5.4-mini）はレビューで発見された
4. **人間の役割はディレクション** — 「刺さる感じがしない」「縦長カードの方がいい」という感性的判断は人間が担う
5. **声が AI を人格にする** — TTS 読み上げにより、AI は単なるツールではなく「チームメイト」として認識される
