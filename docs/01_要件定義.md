# Linny - 要件定義書

## プロジェクト概要

社内LINEもどき「Linny」の開発。
LINEの代替として社内コミュニケーション基盤を自社構築する。

## 背景・課題

- LINEの画像・動画アップロードが個人スマホの容量を圧迫
- 有料オプションが多く、Messenger APIの料金が高額
- 社内システムとの連携コストが高い
- Slackは社内のITリテラシーの都合で導入困難（LINEのUIに慣れている）

## 確定した前提条件

| 項目 | 内容 |
|------|------|
| プラットフォーム | PWA（スマホ+PC両対応） |
| 通信方式 | WebSocket（Socket.IO）。将来のAI push・音声通話の土台 |
| メディア保存 | サーバー側保存（端末容量を使わない） |
| 利用者数 | 約50名（将来拡張見込みあり） |
| インフラ | 自社サーバー（AIエンジン稼働中の環境を利用） |
| 外部公開 | Cloudflare DNS経由で外部アクセス許可 |
| アクセス範囲 | 社外（自宅・外出先）からも利用 |
| 認証 | 初期は社員番号ベース、将来LDAP対応 |
| DB連携 | API/MCP経由、リードオンリーDBユーザでView参照 |
| 権限管理 | LINEと同じグループ単位の管理 |
| AI連携 | 別モジュール、MCP経由で自動応答（Phase 3） |
| UIの方向性 | LINEの完全コピーではなく、LINEライクな操作感があればOK |

## フェーズ定義

### Phase 1（MVP・最優先）

- 1対1チャット / グループチャット
- テキストメッセージ送受信（リアルタイム）
- 画像・動画・ファイルのアップロード/表示（サーバー保存）
- 既読機能（トーク一覧: 未読数、トーク画面: 既読数）
- Push通知（PWA Service Worker）
- 社員番号ログイン（JWT認証）
- LINEライクなUI（吹き出し形式）
- リプライ機能（LINEと同じシンプルな引用形式、ツリーにはしない）
- ユーザー管理画面（管理者ダッシュボード: ユーザー一覧・作成・編集・無効化）

### Phase 2（実用拡張）

- メッセージ検索
- メンション（@通知）
- アルバム機能（画像のグルーピング・閲覧UI）
- スタンプ/絵文字（社内オリジナルも可）— 低優先
- ピン留めメッセージ — 低優先

### Phase 3（将来）

- AI Bot連携（MCP経由のDB参照・自動応答）
- 音声通話 / ビデオ通話（WebRTC）
- LDAP認証連携
- 勤怠連携等の業務Bot

## 技術スタック

| レイヤー | 技術 | 選定理由 |
|----------|------|----------|
| フロントエンド | React + Vite | PWA対応が成熟、Viteで高速ビルド |
| 状態管理 | Zustand | Reduxより軽量、50人規模に最適 |
| バックエンド | Node.js (Express) | WebSocket性能優先、フロントとJS統一 |
| WebSocket | Socket.IO | 成熟したライブラリ、自動再接続・フォールバック |
| DB | PostgreSQL | RLS対応、信頼性 |
| キャッシュ | Redis | 在席状態管理、Socket.IO Pub/Sub |
| ファイルアップロード | multer | Node.jsのデファクト |
| サムネイル生成 | sharp | 高速・メモリ効率良 |
| Push通知 | web-push | Node.js用Web Push送信 |
| リバースプロキシ | Nginx | パスルーティング、メディア配信 |
| コンテナ | Docker Compose | PostgreSQL・Redis統一管理 |
| 外部公開 | Cloudflare | DNS、SSL終端、DDoS防御 |

## 技術スタック選定の経緯

### Node.js vs FastAPI

メッセンジャー本体はNode.js一本とした。理由：

- リアルタイム通信（WebSocket）はNode.jsの方がレスポンスが良い（Webトランシーバー開発の実績から確認済み）
- フロント（React）とバックエンドで言語が統一でき、プロジェクト運営上有利
- AI連携は将来Phase 3でPythonの別モジュールとして疎結合で接続（MCP/REST API経由）
